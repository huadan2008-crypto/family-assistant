# v4.0 · 2026-06-22

这是一次架构级改动：**"录入识别"和"归档整理"彻底解耦**。

## 为什么要做这个改动

之前的版本里，账单录入和文件归档（移动到"会计师交接/账本/季度"文件夹）是绑在一起的——选完账本立刻上传归档。这导致每次调整归档逻辑（比如改文件夹结构、改查重规则），都需要重新拍照、重新上传、重新走一遍 AI 识别，既浪费 token，也浪费你的时间。

v4.0 把这两件事拆成两个独立阶段：

- **录入阶段**：拍照/上传 → AI 识别 → 选账本 → 数据写入数据库 + 文件上传到 Drive **暂存文件夹**（不分账本）
- **归档阶段**：随时可以打开「📁 待归档」页面，批量把暂存文件夹里的文件**移动**（不是重新上传）到对应账本的「会计师交接」文件夹

这样归档逻辑以后再怎么改，都是对已有数据/已有文件的重新处理，不会要求你重新录入。

## ✨ 新增：「📁 待归档」管理页面

- 财务页新增按钮，显示待处理数量（比如「📁 待归档 (5)」）。
- 列出所有还没归档的账单：商家、日期、金额、当前账本，每行可单独调整账本。
- **批量设置账本**：勾选多笔，统一选一个账本，一键应用到所有勾选项——这是你最初提出方式二时想要的效果。
- 账本仍为「待定」的账单会用醒目颜色标出，提示"需要先决定账本才能归档"，点「归档已勾选项」时会自动跳过这些（不会被错误归档），其余正常处理。
- 归档完成后会有统计提示：成功几笔、跳过几笔（待定）、失败几笔（如果失败，控制台会有详细原因）。

## ✨ 新增：Drive 文件移动机制（不重新上传）

- 新增 `moveFileToAccountingFolder()`，用 Google Drive API 的 `addParents`/`removeParents` 更新文件所属文件夹，这是一个轻量的元数据操作，不涉及重新读取或传输文件内容。
- 文件已经在"收据归档/年月/"暂存文件夹里（这一步在录入阶段就完成了），归档时只是把它"剪切粘贴"到「会计师交接/账本/季度/」。
- 如果某账单改成了"个人"等不需要凭证的账本，会直接标记为完成，文件留在暂存文件夹不动（不需要也不会被移动到会计师文件夹）。

## ✨ 调整：录入阶段不再立即归档

- `saveBill`、`riFallbackSingle`、`confirmReceiptItems` 三处入口都改为：保存数据时只标记 `archiveStatus: 'pending'`，不再触发任何文件上传/移动操作。
- 新增账单字段：
  - `archiveStatus`：`'pending'`（待归档）或 `'archived'`（已完成），只有真正关联了暂存文件的账单才会有这个字段
  - `driveFileId`：指向 Drive 暂存文件夹里那份原始文件的 ID，归档时用它来定位要移动的文件
- 账单列表里，待归档的账单会显示「📁 待归档」小标记，方便你一眼看出还有哪些没整理。

## ✨ 调整：一张收据拆分给多个账本时的归档处理

- 如果一张收据被你拆成了多个账本（比如超市小票里既有公司用品又有个人采购），Drive 上只有一份原始文件，不可能同时移动到两个文件夹。
- 处理方式：**金额最大的那个分组**持有归档权（标记为待归档、关联文件ID），其余分组只记录 `receiptId` 作参考链接，不会重复出现在待归档列表里，也不会触发多次移动。

## 🔒 未改动

- Google OAuth、Drive sync 机制不变。
- 「收据归档/年月/」这个全量暂存文件夹结构不变，继续作为所有收据的第一落脚点。
- 「用作凭证」按钮（没有原始收据时，用信用卡账单本身生成替代凭证）逻辑不变——这个场景文件是凭空生成的，不存在"先暂存再移动"的需求，所以仍是确认后立即生成并上传。
- PDF 信用卡账单的归档保持原来的"按年月"结构，不受这次改动影响（之前已确认：信用卡月结单天然跨账本，不适合拆分按账本归档）。
- 商家账本记忆表、分类默认账本、消费趋势、严格查重等 v3.13-v3.15 功能全部保留，不受影响。

## 💡 使用建议

1. 平时正常录入账单（拍照收据、CSV/PDF导入、手动记账），不需要额外操作。
2. 攒了一批之后（比如月底/季度末），打开「📁 待归档」，先处理"待定"账本的那些（批量选好账本），再点「归档已勾选项」。
3. 如果归档过程中网络不稳定导致部分失败，失败的账单会保留 `archiveStatus: 'pending'`，留在列表里，下次重新打开「待归档」点一次就行，不会丢失、也不会重复创建数据。
