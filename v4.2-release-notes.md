## v4.2 · 2026-06-22

### 🐛 修复：Excel (.xlsx) 文件上传报错"Unexpected token... is not valid JSON"（严重bug）

**问题**：上传 `.xlsx` 格式的信用卡账单后，报错"CSV/Excel 解析失败：Unexpected token '我'..."，本质是AI返回了一段说明性文字而不是预期的JSON数据。

**根本原因**：`.xlsx` 文件本质是一个 ZIP 压缩包（内部是XML结构），但之前的代码用 `file.text()` 把它当**纯文本文件**读取——这样读出来的是一堆乱码二进制内容，根本不是Excel表格里的文字。把这堆乱码发给AI识别，AI自然无法从中提取出账单数据，只能回复"我无法从提供的内容中..."这样的说明文字，而代码没有检查这种情况，直接尝试 `JSON.parse()`，于是崩溃报错。

**用真实文件验证**：用你上传的 Amex 信用卡 Excel 账单（包含 "Transaction Details" 和 "Transaction Summary" 两个sheet，且前6行是账户摘要信息、真正的表头在第7行）测试，确认问题完全复现，且修复后能正确提取出全部28笔交易。

**修复方案**：
1. **引入 SheetJS**（业界标准的纯前端Excel解析库），正确读取 `.xlsx`/`.xls` 的二进制结构，得到真正的表格行列数据——而不是乱码文本。
2. **完全本地解析，不再需要AI参与**：Excel 现在跟 CSV 一样走本地JS直接解析，不消耗任何 Anthropic API token，速度也更快，不再有"AI识别失败"这一类不确定性。
3. **智能识别真正的表头行**：新增 `findHeaderRowIndex()`，自动扫描前20行，找到同时包含"日期"和"金额"类关键词的那一行作为表头——这样即使像你这份Amex文件一样，前几行是账户摘要信息（卡名、户主、账号）夹在数据前面，也能正确跳过，定位到真正的表头。
4. **自动挑选正确的工作表**：如果Excel里有多个sheet（比如你这份文件的"Transaction Details"和"Transaction Summary"），优先选名字像"明细/Detail/Transaction"的那个，避免误读汇总表。
5. **CSV和Excel共用同一套提取逻辑**：新增 `extractTransactionsFromRows()`，两种格式解析出来的二维数组结构一致，用同一套列名识别规则（date/note/amount/hst/cat/ledger），减少了重复代码，也意味着以后CSV那边的解析改进（比如新增列名别名）会自动同时惠及Excel。

### 🐛 顺带修复：CSV文件名大小写判断问题

- 之前判断文件是否为CSV用的是 `file.name.endsWith('.csv')`，区分大小写——如果文件名是 `.CSV`（大写）会被误判为不是CSV，走错处理路径。
- 改为 `file.name.toLowerCase().endsWith('.csv')`，不再区分大小写。

### 🔒 未改动
- Google OAuth、Drive sync、`appState.bills` 数据结构完全未变。
- CSV 的解析逻辑本身没有变化（列名识别规则、日期解析、查重等），只是和 Excel 共用了底层的"从二维数组提取交易"这部分代码。
- PDF 账单解析（走AI文档理解）不受影响，因为PDF本身确实需要AI来读取版面内容，跟这次修的"Excel被误当纯文本"是两个不同的问题。

### 💡 关于 token 成本
这次修复还有一个附带好处：之前 Excel 文件走 AI 解析，每次上传都要消耗 token；现在改成本地直接解析，**Excel 导入和 CSV 一样完全免费**（不调用 Anthropic API）。之前我们讨论过"CSV比PDF省token"，现在这个结论同样适用于 Excel 了。
