## v3.12 · 2026-06-22

### 🐛 修复：CSV导入日期解析失败，账单被错误归到同一个月（严重bug）

**问题**：上传信用卡账单CSV后，所有交易的日期没有被正确识别，结果多笔不同日期的账单被错误地堆到了同一个月。

**根因排查发现了两个叠加的 bug**：

1. **日期格式识别过窄**：原有逻辑先用一个只认纯数字格式（如 `2026-03-21`、`03/21/2026`）的正则做前置筛选，筛选不过直接放弃，根本不会再尝试用更宽松的方式解析。但很多信用卡账单（比如 Amex）导出的日期格式是 `21 Mar 2026`（日 + 英文月份缩写 + 年），完全不匹配那个正则，导致日期始终解析失败、字段为空。

2. **CSV换行符切分错位**（更隐蔽的连带问题）：原有代码用最简单的 `text.split('\n')` 按行分割CSV。但标准CSV格式里，被引号包裹的字段（比如地址"OTTAWA\nON"）内部可能包含换行符，这种换行不应该被当作"换行到下一笔交易"。按行硬切会把这类地址字段从中间切断，导致后面所有列整体错位——本该是"日期"的位置读出来变成了地址碎片（如"ON"），看起来就像日期完全消失了。

**修复**：
- 新增 `parseCSVText()` 函数：用逐字符状态机正确解析CSV，识别"引号内的换行/逗号不是真正的分隔符"，避免地址等字段导致行错位。
- 新增 `parseFlexibleDate()` 函数：不再用正则前置过滤，直接尝试用更智能的方式解析日期——
  - 先处理"日在前"的纯数字格式（如 `21/03/2026`，根据第一段数字>12判断是日不是月，避免和 MM/DD/YYYY 弄混）
  - 再交给浏览器原生 `Date` 解析，能正确处理 `21 Mar 2026`、`Mar 21, 2026`、`2026-03-21`、`03/21/2026` 等绝大多数常见格式
  - 实在解析不出来就留空，导入审核界面会用"日期未知"明确提示，不会再悄悄归错月份

**已用真实账单CSV验证**：35笔交易全部正确识别日期和金额，无一遗漏。

### 🔒 未改动
- Google OAuth、Drive sync、`appState.bills` 数据结构完全未变。
- CSV列名识别规则（date/note/amount/hst/cat/ledger等）不变，只是底层切分和日期解析更健壮。
- PDF、Excel、收据拍照的解析路径不受影响。

## 关于 CSV vs PDF 解析的 token 费用对比

**CSV 解析明显更省钱**，原因：

- **CSV走的是本地JS代码解析**（这次修的就是这部分逻辑），完全不调用 Anthropic API，**零token消耗**。
- **PDF走的是AI视觉/文档理解**（`callClaudeAPI` 调用 `claude-sonnet-4-6`，把整份PDF当文档发给模型读取并提取交易），这部分**按输入+输出token计费**，PDF页数越多、交易越多，消耗越大。
- 唯一例外：**Excel文件**也是走AI解析（因为Excel格式比CSV复杂，本地解析不稳定），所以Excel导入的token消耗和PDF类似，CSV是三种格式里唯一完全免费（不耗API额度）的。

**实际建议**：如果信用卡/银行支持导出CSV格式（大多数都支持），优先选CSV而不是PDF或Excel——不仅免费，而且这次修复后CSV的日期、金额解析准确度也有保障。PDF适合那种没有CSV导出选项、只能拿到账单图片或扫描件的情况。
